## 示例 6-2：模式 8（外部系统对接 — 渠道对接场景）

**用户输入**：订单系统通过 API 对接悟空租车平台

**大纲（外部对接类结构）**：
```markdown
## 需求大纲

### 问题陈述
马达系统需要接入悟空租车平台，实现门店、车型、订单全链路对接，拓展分销渠道。

### 目标
1. 门店信息、车型商品可同步至悟空平台
2. C端用户可在悟空平台完成询价、下单、支付、取还车全流程
3. 押金、违章扣款通过平台代扣完成

### 非目标
1. 不改造马达现有订单流程 | 理由：悟空对接在马达外层包装，不侵入核心逻辑
2. 不做悟空后台管理页面 | 理由：使用悟空已有后台，马达只提供 API 能力

### 功能模块
1. 门店信息/政策（网点新增/更新、服务圈、异地取还）
2. 车型车辆（标准车型、价格设置、车辆新增/更新/上下架）
3. 订单流程（询价下单、押金、取还车、修改订单、续租）
4. 兼容现有功能（工单、开票、违章、导出；最终写入详细需求，不单独成章）

### 待确认（归属标注）
1. 押金扣款是否支持线下入口？（产品决策）
2. 悟空已有的门店/车辆老数据如何导入？（业务决策）
3. 换车接口是否拦截所有平台？（技术决策）
```

**详细需求写法（接口契约示例）**：
```markdown
### 5.2 车型价格配置

### 功能描述
a. 渠道车型商品需单独配置悟空平台价格
b. 价格分为基础保障（平日/周末）和特殊日期价格
c. 价格保存后调用【网点车型基础价格新增/更新】API 同步平台

### 接口契约
**【网点车型基础价格新增】API**

| 字段 | 说明 | 枚举/规则 |
|------|------|-----------|
| vender_code | 供应商编码 | 必填 |
| site_id | 网点ID | 必填 |
| car_type_id | 车型ID | 必填 |
| day_price | 日租金 | 5位以内整数 |
| hour_price | 时租金 | 日租金/7 = 时租金 |
| poundage_amount | 手续费 | ≤35 整数 |
| car_deposit | 租车押金 | 20万以内整数 |
| illegal_deposit | 违章押金 | 500~2000 整数 |
| basic_insurance | 基础保障价格 | 按车型有上下限（见限价表） |

**校验规则**：
- 日租金 5 位以内整数，提示"租金请输入5位以内的整数"
- 手续费不超过 35，提示"手续费需要为最大不超过35的整数"
- 违章押金范围 500~2000，提示"违章押金需要500~2000元区间"
- 保险价格按车型有不同上下限（如 SUV_D 基础保障 min:80, max:120）
```

## 验收用例

每个 PRD 末尾必须包含验收用例，按页面维度组织，写清楚验收时的操作步骤。

**格式**：

```markdown
## N. 验收用例

### N.1 XX页面验收

| 序号 | 验收项 | 操作步骤 | 预期结果 |
|------|--------|---------|---------|
| 1 | 筛选项展示 | 打开XX列表页，查看筛选区域 | 新增的XX筛选项正确展示，默认不选择 |
| 2 | 筛选功能 | 选择XX筛选项，点击查询 | 列表正确过滤，只展示符合条件的数据 |
| 3 | 列表展示 | 查看列表各列 | 新增列正确展示数据，无数据时显示 `-` |
| 4 | 操作按钮-正常 | 点击XX按钮 | 弹框/页面正确打开，操作成功后toast提示 |
| 5 | 操作按钮-异常 | 不填必填项，点击确定 | toast提示缺少必填项 |
```

**验收用例编写原则**：
- 每个页面的每个功能点至少 1 条验收用例
- 操作步骤写 PM 的实际操作（打开页面、点击按钮、输入数据），不写技术实现
- 预期结果写用户可感知的反馈（展示正确、toast 提示、页面跳转），不写系统内部逻辑
- 覆盖正常路径 + 异常路径（空数据、必填项缺失、边界值）
- 如果涉及状态流转，覆盖每个状态的入口和出口
