# 步骤 1：解析输入

> 加载时机：工作流第一步，用户描述需求后，用于解析输入来源和提取结构化信息。

用户描述需求后，先检测输入来源，再提取结构化信息，判断需求类型。

## 1-walle. 瓦力需求检测（优先判断）

检查用户输入是否包含瓦力需求编号（`#数字` 或纯数字，如 `#638101`）。

**如果有瓦力编号**，执行以下流程：

### 1-walle-a. 从瓦力拉取需求信息

调用瓦力 MCP 工具获取完整需求上下文：

| 调用 | 用途 |
|------|------|
| `get_issue_detail(issueId)` | 需求标题、描述（HTML）、优先级、产品经理、发起方 |
| `get_issue_document(issueId)` | 已有 PRD 文档内容（可能为空） |
| `get_issue_summary(issueId)` | AI 摘要（可能为空） |
| `get_issue_relations(issueId)` | 前置/后置/关联需求 |

### 1-walle-b. 解析需求描述

瓦力需求描述通常是 HTML 格式，按以下结构拆解：
1. **识别已取消项**：HTML 中 `<s>`、`<del>`、`<strike>` 标签包裹的内容为已取消的需求项。标注为"已取消"，**不纳入 PRD 范围**，但在大纲中列出供产品确认（"以下需求项已删除线标记，确认不纳入 PRD？"）
2. **提取【需求背景】**：业务上下文和动机
3. **提取【需求价值】**：做这件事的理由
4. **提取【需求详情】中的每个改动点**：逐条拆为独立需求项（跳过已取消项）
5. **提取图片链接**：**禁止用 Read 工具查看图片文件**（会触发 400 API 错误）。图片链接仅作为"原型参考"记录到 PRD 中，请用户口述图片内容（页面布局、改动标注等关键信息）
6. **提取元数据**：优先级、发起方、业务期望完成时间、产品线
7. **提取沟通进展**：`最新沟通进展` 和 `最新沟通进展更新日期` 字段，作为需求变更上下文（可能包含需求调整原因、评审结论等关键信息）

### 1-walle-c. 评估信息完整度

| 信息等级 | 判断标准 | 处理方式 |
|---------|---------|---------|
| **充足** | 有明确的改动点描述、业务背景、涉及页面 | 直接采用，标注 `[来自瓦力]` |
| **部分缺失** | 改动点清楚但缺少具体字段/控件/枚举/校验规则 | 针对缺失项追问 |
| **严重不足** | 只有一句话标题，没有详细描述 | 退化为"从零追问"模式 |

**瓦力需求描述的常见信息分布**（实测规律）：
- ✅ 通常有：业务背景、价值、具体改动点、截图
- ❌ 通常缺：字段控件类型、枚举值列表、校验规则、展示条件的精确逻辑、权限要求
- 信息充足度约 **60-70%**，够启动 PRD 生成，但关键细节仍需追问

### 1-walle-d. 分支处理

- 信息充足/部分缺失 → 带着瓦力数据进入步骤 2（模式识别）
- 严重不足 → 先追问用户补充需求描述，再继续
- 瓦力 `get_issue_document` 返回了已有 PRD → 询问用户："已有 PRD 文档，是要在此基础上补充完善，还是重新生成？"

---

**如果没有瓦力编号**，继续以下流程。

## 1a. 提取关键词

从用户输入中提取：
- **业务对象**：采购单、需求单、订单、物资...
- **动作**：增加筛选、新增字段、修改状态、批量导出...
- **条件**（如果有）：当 XX 时、XX 状态下...

## 1b. 判断需求类型

| 类型 | 特征 | 后续策略 |
|------|------|---------|
| **迭代** | 修改已有功能（"加筛选""加字段""改状态"） | Hub 深度查询，精准改动 |
| **新模块** | 从零创建（"做一个XX管理""新增XX模块"） | 需要领域建模，提问量大 |
| **外部对接** | 系统间数据交互（"对接XX""调用XX接口"） | 关注数据流向和异常处理 |

需求类型决定：
- 系统感知的深度（迭代要深查，新模块可能没东西可查）
- 追问策略（迭代预填多问得少，新模块预填少问得多）
- PRD 结构（新模块需要领域建模章节，外部对接需要数据流说明）

## 1b-2. 模式特殊处理规则

部分模式有额外的覆盖要求：

**模式 10b（审批表单字段变更）**：
- 区分只读展示字段和可编辑字段
- 标注每个字段的数据来源（从订单获取 / 从退款记录获取 / 系统计算）
- 标注格式化规则（日期格式 / 金额格式 / 租期格式）

**模式 11（通知/消息触发）需要覆盖**：
- 触发时机（什么事件触发通知）
- 通知渠道（短信/邮件/App 推送/站内信）
- 接收人（谁收到通知）
- 内容模板（通知写什么）
- 发送失败处理（异步还是同步，失败是否重试）

**模式 12（附件/上传变更）需要覆盖**：
- 文件类型限制（支持哪些格式）
- 数量上限（最多几张/几个）
- 大小限制（单个文件最大多少）
- 格式要求（命名规则、分辨率等）
- 预览方式（是否支持在线预览）
- 历史数据处理（已有附件是否需要迁移）

## 1b-3. 跨模块需求处理

当需求涉及多个模块时（如"应急收车创建第三方运输需求单"涉及应急收车 + 需求单 + 采购单）：
- 对每个模块分别执行系统感知
- 追问时按模块分组提问，不让用户在不同模块间反复切换
- 输出时按模块拆分详细需求表格（每个模块一个 `5.X` 小节）
- 在模块之间标注联动关系

## 1c. 简单需求快速通道

如果需求足够简单（1 个模式 + Hub 有数据 + 用户描述清晰），步骤 1-3 的内部分析可以快速完成，直接进入步骤 4 输出需求大纲。不需要向用户展示分析过程，只展示大纲结果。
