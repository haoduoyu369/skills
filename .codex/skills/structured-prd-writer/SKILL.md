# 结构化需求解析 Skill

## Bundled references

本技能已随项目级技能继承 `references/style-guide.md`。当目标项目根目录没有 `docs/style-guide.md` 时，读取该 bundled reference 作为设计规范兜底依据。

## 定位

write-spec 擅长「从一个想法出发生成 PRD」，但当输入本身已经是结构化需求文本时，它会将多层级的规则链「拍平」进一个槽位，丢失了：

- 功能入口的独立性（一个需求可能包含多个独立入口）
- 规则之间的父子关系（校验链、条件分支）
- 互斥条件 vs 并行条件（三种换车原因是互斥的，但每种原因下的校验是串行的）

本 skill 在 write-spec 的「理解需求」和「生成 PRD」之间插入 **文本层级解析** 阶段，解决上述问题。

## 工作流程

### Phase 0: 文本层级解析（新增核心步骤）

收到需求文本后，不直接进入 write-spec 流程，而是先做结构化拆解。

#### Step 0.1 — 层次识别

扫描输入文本，按语义标签将内容分配到以下层级：

| 层级 | 标签 | 说明 | 示例 |
|------|------|------|------|
| L0 | 背景层 | 为什么要做，当前痛点 | 「免费升级被滥用，中台无法管控」 |
| L1 | 意图层 | 要改什么，功能入口 | 「取车前免费升级 + 中途换车两个入口」 |
| L2 | 修改点层 | 每个入口的具体改动 | 「改排车页面增加车型级别展示」 |
| L3 | 规则链层 | 校验规则、条件分支、阻断逻辑 | 「选择车辆损伤→关联车损单→判断待维修→阻断提示」 |
| L4 | 交互层 | UI 呈现、弹窗、移动端适配 | 「弹窗选择车损记录」 |

输出格式：每个层级一个段落，层级之间用缩进表示嵌套关系。

#### Step 0.2 — 功能入口识别

从 L1 意图层中提取**独立功能入口**。判断标准：

- 如果两个改动对应不同的操作流程（如「取车前免费升级」和「中途换车」），则标记为两个独立入口
- 如果一个入口是另一个的子流程，则标记为父子入口
- 每个入口独立输出一组用户故事

输出格式：

```
入口 A：免费升级管控
  ├── 涉及页面：改排车 → 选择车型 → 提交申请
  └── 校验规则：[列表]

入口 B：中途换车合规校验
  ├── 涉及页面：发起换车 → 选择原因 → 关联单据 → 选择车型 → 提交
  └── 校验规则：[列表]
```

#### Step 0.3 — 规则分支树提取

对每个功能入口下的 L3 规则链，构建规则分支树。

规则分支树必须包含以下要素：
- **触发条件**：什么操作触发该规则
- **校验步骤**：按顺序执行的校验
- **分支条件**：if-else 分支
- **互斥标记**：标注哪些分支是互斥的
- **阻断条件**：什么情况下阻止提交，提示文案

输出格式（树形文本）：

```
规则入口：中途换车 - 选择换车原因
│
├── [互斥分支 ①] 车辆损伤
│   ├── 校验：查询当前订单关联车损单
│   ├── 分支：有车损单且状态=待维修
│   │   └── 自动带出 → 可提交
│   ├── 分支：无关联车损单
│   │   └── 弹窗选择待维修/维修中车损单 → 可提交
│   └── 阻断：无任何有效车损记录
│       └── 提示：「当前车辆无有效车损记录，无法发起车损换车」
│
├── [互斥分支 ②] 车辆保养
│   ├── 校验：查询车辆保养预警记录
│   ├── 分支：有保养预警记录
│   │   └── 自动关联 → 可提交
│   └── 阻断：无保养预警记录
│       └── 提示：「当前车辆无符合条件的保养预警记录，无法发起保养换车」
│
└── [互斥分支 ③] 其他原因
    ├── 填写备注信息
    ├── 校验：同级别库存=0
    │   ├── 通过 → 可跨组选择车型 → 可提交
    │   └── 阻断：同级别仍有库存
    │       └── 提示：「当前车型级别组仍有库存，请重新选择车辆」
```

#### Step 0.4 — 校验规则汇总表

将所有 L3 规则提取为汇总表，列结构：

| 校验点 | 触发场景 | 校验逻辑 | 通过条件 | 阻断提示 |
|--------|---------|---------|---------|---------|

### Phase 1-5: 沿用 write-spec 流程

文本层级解析完成后，将结果输入 write-spec 的标准流程：

1. **理解需求** → 使用 Phase 0 的拆解结果
2. **收集上下文** → 对话式追问（同 write-spec）
3. **拉取上下文** → 从连接工具搜资料（同 write-spec）
4. **生成 PRD** → 按 write-spec 7 章模板输出，但每个功能入口独立写一组用户故事和需求
5. **评审迭代** → 确认调整

## PRD 输出增强规则

### 用户故事分组

每个独立功能入口生成一组用户故事，避免混在一起：

```markdown
## 用户故事

### 入口 A：免费升级管控

- As a 门店员工, I want ...
- As a 运营中台, I want ...

### 入口 B：中途换车合规校验

- As a 门店员工, I want ...
- As a 区域经理, I want ...
```

### 需求条目编号

需求编号包含入口标记，一眼看出归属：

```markdown
| P0-A1 | 免费升级 | 提交时校验同级别库存 | ...
| P0-A2 | 免费升级 | 同级别无库存时允许跨组 | ...
| P0-B1 | 中途换车 | 选择换车原因 | ...
| P0-B2 | 中途换车 | 车辆损伤-关联车损单 | ...
| P0-B3 | 中途换车 | 车辆保养-关联保养记录 | ...
| P0-B4 | 中途换车 | 其他原因-库存校验 | ...
```

### 规则树附录（仅限 Phase 0-5 分析文档）

结构化需求分析文档末尾可附上完整的规则分支树，用于产品和业务评审。Phase 6 研发版 PRD 固定 6 个模块，不新增附录章节；必要规则收敛到"详细需求"对应功能点说明中。

```markdown
## 附录：规则分支树

（粘贴 Phase 0.3 的完整规则树）
```

## Phase 6: 研发版 PRD 模板（固定 6 模块）

### 背景

当产品需要输出给研发的标准需求文档时，需遵循当前产品方案设计流程的研发版 PRD 模板。最终文档只包含 6 个一级模块：版本管理、需求背景、流程图、功能清单、详细需求、验收用例。

### 模板结构

```markdown
# {产品名称} v{版本号}

## 1. 需求版本管理
| 版本 | 日期 | 修改人 | 修改内容 |
|------|------|--------|---------|
| V1.0 | YYYY-MM-DD | {修改人} | 评审版本 |

## 2. 需求背景
### 2.1 业务背景
- 说明当前业务背景、业务痛点和不做的影响。

### 2.2 业务目标
- 说明本次需求要达成的业务目标和成功标准。

## 3. 流程图
![{流程图名称}](../diagrams/{编号}-flow.png)

### 3.1 流程说明
a. {流程起点、触发条件}
b. {关键判断分支和用户/系统动作}
c. {流程终点、成功/失败结果}

## 4. 功能清单
| 模块 | 功能 | 功能点 | 变更类型 |
|------|------|--------|---------|

## 5. 详细需求
### 5.1 {页面/功能模块}
| 功能点 | 原型 | 说明 |
|--------|------|------|

## 6. 验收用例
| 序号 | 验收项 | 操作步骤 | 预期结果 |
|------|--------|---------|---------|
```

版本规则：评审时为 `V1.0`；评审通过后修改为 `V1.1`；后续开发、测试提 bug 单导致需求调整时继续按 `V1.2`、`V1.3` 递增，并记录修改内容。

### 适用场景

- 需求已通过 structured-prd-writer 完成层级解析和规则提取后
- 需要输出固定 6 模块研发版 PRD 给研发和测试团队
- 需要与现有研发团队的文档模板保持一致

### 与 Phase 0-5 的关系

Phase 0-5 输出的是「结构化需求分析文档」（面向产品和业务），Phase 6 输出的是「研发版 PRD」（面向研发和测试）。两者互补：
- Phase 0-5 文档：保留完整规则分支树、需求条目编号、用户故事，用于需求评审和逻辑验证
- Phase 6 文档：精简为固定 6 模块研发版 PRD，保留背景、流程、功能清单、详细需求和验收用例，用于研发实现和测试案例编写

## Phase 7: 前端代码原型生成（v3.0 前端代码驱动）

### 背景

Phase 0-6 产出了完整的需求文档和技术规格。Phase 7 将需求方案直接转化为前端 Vue 代码修改——通过 gitnexus-hub 获取 Vue 源码，放入 ProtoHub-mada 的独立 Vue 预览项目（`preview/`）中修改，用 `npm run dev` 构建预览。不依赖前端仓库 Git 权限。

### 前端代码目录

```
ProtoHub-mada/preview/                 # 独立 Vue 预览项目（产品团队自主维护）
├── src/views/                         # Vue 页面源码（从 Hub 获取 + 修改）
│   ├── sale/order/                    # 从 Hub 获取的已有页面
│   ├── prototype/                     # 原型专属目录（新增页面放这里）
│   └── ...
├── src/components/                    # 共享组件（从 Hub 获取后放入）
├── package.json                       # Vue 3 + Element Plus（版本对齐线上）
├── vite.config.ts
└── .env.prototype                     # 原型配置
```

### 生成流程

#### Step 7.1 — 确认预览项目就绪

```bash
# ProtoHub-mada 的 preview/ 是独立 Vue 预览项目
cd preview
ls -la                    # 确认目录结构完整
npm install               # 首次使用或依赖更新后执行
```

> Vue 源码通过 gitnexus-hub 获取，不需要前端仓库 Git 权限

#### Step 7.2 — 读取目标 Vue 文件

通过术语映射表定位前端页面路径，然后读取目标 `.vue` 文件：

```bash
# 从 Hub 获取 Vue 源码（用于了解现有结构）
python3 scripts/hub_client.py context --repo "{仓库名}" \
  --uid "File:{组件路径}" --content

# 读取 ProtoHub 仓库中的 Vue 文件（用于直接修改）
cat preview/src/views/{页面路径}/index.vue
```

#### Step 7.3 — 修改/编写前端代码

直接修改前端仓库中的 Vue 文件：
- 新增/修改/删除字段、按钮、列等 UI 元素
- 使用前端项目已有的 Element Plus 组件和共享组件（直接 import，不需要 CDN）
- 样式以现有 Vue 源码的 `<template>` + `<style scoped>` 为唯一标准
- 禁止生成独立 HTML 文件代替修改 Vue 代码

#### Step 7.4 — 构建预览

```bash
cd frontend
npm install        # 首次需要安装依赖
npm run dev        # 启动开发服务器预览
```

在浏览器中查看修改后的页面，确认变更效果。

### 样式合规检查

修改后必须检查：
- [ ] 新增元素与源码中最相似元素使用相同的 Element Plus 组件
- [ ] 新增元素的 props 与同级元素一致
- [ ] CSS 类名来自源码 `<style scoped>`，无自定义类
- [ ] 无标注装饰（无红色高亮、角标、删除线等）
- [ ] `npm run dev` 构建通过，无编译错误

### 样式对齐工作流

生成原型前，通过 Hub 获取前端代码样式：

1. 通过 Hub 查询目标页面 Vue 源码（`hub_client.py context --content`）
2. 提取 `<template>` 中的组件结构（el-table / el-form / el-descriptions 等）
3. 提取 `<style scoped>` 中的自定义 CSS 类名和样式
4. 识别共享组件（SearchForm / SearchItem / Table）的渲染结构
5. 修改 Vue 代码时忠实复现以上结构

## 实际案例参考

### 案例：车辆空驶预警处理与中台稽查系统

**背景**：门店车辆移动稽查系统经过多轮迭代（v1.0 → v1.8），形成了完整的「预警→处理→稽查→定责」闭环。

**Phase 0 层级解析关键点**：

| 层级 | 内容 |
|------|------|
| L0 背景 | 公车私用风险、车辆资产管理漏洞 |
| L1 入口 | A-空驶预警（自动）、B-门店处理（店长）、C-中台稽查定责（运营） |
| L2 修改点 | 双通道触发（围栏+轨迹）、门店统一入口、稽查列表/详情/看板 |
| L3 规则链 | R15 前置校验（业务单覆盖）、R18 双通道触发（围栏>3km OR 轨迹>5km）、R16 双轨异步 |
| L4 交互 | 门店：空驶预警面板（唯一入口）；中台：稽查列表+详情+看板+试点配置 |

**R18 双通道触发规则树（Phase 0.3 示例）**：

```
[双通道触发 R18] 空驶预警触发前
    │
    ├── 前置条件（须全部满足）
    │   ├── 车辆状态 = 空闲
    │   ├── 无进行中的订单
    │   ├── 无关联有效业务单/任务工单
    │   └── 门店 is_pilot = 1
    │
    ├── 通道1【围栏触发·优先·早预警】
    │   ├── Haversine(当前坐标, 围栏中心点) > 3,000m → 触发
    │   └── 特点：车辆刚驶出围栏即可感知
    │
    ├── 通道2【轨迹触发·兜底】
    │   ├── SUM(GPS相邻点距离) > 5,000m → 触发
    │   └── 适用：无围栏区域（新店/临时点）
    │
    └── 去重：同一空驶事件仅触发一次，围栏优先
```
