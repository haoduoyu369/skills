# 步骤 4-6：需求大纲 + 产品方案 + 研发版 PRD 生成

> 加载时机：步骤 4（大纲确认）、步骤 5（产品方案）和步骤 6（研发版 PRD 生成）时加载。研发版 PRD 固定为 6 个模块：版本管理、需求背景、流程图、功能清单、详细需求、验收用例。

## 4c. 追问决策矩阵

**核心原则：提问带答案**。每一个问用户的问题，AI 都应该先做过功课——查过 Hub、研究过通用方案、分析过公司惯例。问题是"我认为是 X，你确认吗？"而不是"你觉得是什么？"。

| 条件 | 动作 | 示例 |
|------|------|------|
| Hub 有数据 | 直接填入，标注 `[系统]`，不问 | Hub 查到枚举值 → 直接列入详细需求 |
| 代码中可查到 | 先查 Hub，带结果确认 | "Hub 查到车型组有：经济型/舒适型/SUV，完整吗？" |
| 公司惯例明确 | 填入并标注 `[推断]`，不问 | 枚举筛选项 → 默认多选 |
| 通用最佳实践 | 调研后给出推荐 | "推荐模糊搜索（行业惯例），如需精确请告知" |
| 有多种合理方案 | 给推荐+理由 | "推荐 A（原因：XX），也可选 B（差异：YY）" |
| 只有用户才知道 | 必须问，带选项 | "审批人来源：A.主管列表 / B.手动选择 / C.按金额分级？" |
| 完全不了解 | 先说明查了什么，再问 | "Hub 中未找到相关枚举，请提供选项列表" |

**追问原则**：
- 追问轮数和问题数量根据需求复杂度动态调整：
  - 简单需求（1 个模式）：**1 轮，最多 3 个问题**
  - 中等需求（2-3 个模式）：**1 轮，最多 5-7 个问题**
  - 复杂需求（4+ 模式或多模块）：**最多 2 轮，每轮 5-8 个问题**
- 每个问题都必须 **带推荐答案**（"推荐 A"或"查到是 X"）
- Hub 能查到的 **先查再确认**，不直接问
- 通用模式能推断的 **先推断再确认**，不直接问
- 信息不足时宁可标注 `[待补充]` 也不编造

## 4d. 不同需求类型的大纲要点

大纲的"推断"层需要根据需求类型包含不同内容。注意：这些内容用于前置确认和方案设计，最终 PRD 不新增一级章节，统一收敛到固定 6 个模块中。

| 类型 | 大纲中必须覆盖的要点 |
|------|-------------------|
| **迭代** | 改哪个页面、改什么元素、改前什么样、改后什么样 |
| **新模块** | 业务流程（状态有哪些、怎么流转）、领域概念（有无类型矩阵）、包含哪些页面、每个页面的核心字段 |
| **外部对接** | 数据流向（谁推谁）、触发时机、涉及哪些接口、每个接口的核心字段和枚举、字段映射关系、校验规则、对接失败怎么办 |

**新模块类需求的特殊处理**：

新模块在 Hub 中没有数据，预填量少，提问策略不同：
1. 先问高层结构："这个模块包含哪些页面？（列表页 / 详情页 / 新建弹框 / 编辑弹框）"
2. 建议参考现有模块："参考采购单的模式，你的物资列表需要类似的字段吗？"
3. 用领域概念矩阵辅助："这类业务有哪些类型/分类？需要穷举所有组合吗？"
4. 可以分多次确认——先确认页面结构，再逐页确认字段

**外部对接类需求的特殊处理**：

外部对接类 PRD 的核心交付物是**接口契约**（字段表 + 枚举 + 校验规则），不是页面 UI：
1. 先确认对接范围："涉及哪些功能模块？（门店/车型/订单/押金/库存）"
2. 每个模块确认接口清单："这个模块涉及哪些接口？哪些是主动调用、哪些是被动接收？"
3. 每个接口确认字段表："这个接口有哪些字段？哪些必填？有哪些枚举值？"
4. 确认异常处理："接口超时/失败怎么办？数据不一致怎么补偿？"
5. 确认兼容点："需要兼容哪些现有功能？（工单/开票/导出/违章）"

## 5a. 固定 PRD 结构

用户确认大纲和产品方案后，生成研发版 PRD。

研发版 PRD 只允许以下 6 个一级模块，按固定顺序输出：

``` 
1. 需求版本管理
2. 需求背景
3. 流程图
4. 功能清单
5. 详细需求
6. 验收用例
```

**严禁新增一级模块**：
- 不单独输出"问题与目标/需求目的/非目标"
- 不单独输出"业务流程/领域概念/数据流向/异常处理/兼容现有功能"
- 不单独输出"开放问题/待确认/来源标注/附录"

**内容收敛规则**：
- 业务目标、当前痛点、做成标准 → 写入"需求背景"
- 业务流程、状态机、数据流向、外部系统交互 → 写入"流程图"，并配图片和文字描述
- 领域概念、规则矩阵、接口契约、异常处理、兼容影响 → 写入"详细需求"对应功能点说明
- 非目标、开放问题、来源标注 → 只保存在 `.tmp/solution_brief.md` 或 `.tmp/prd_traceability.md`，不进入研发版 PRD

## 5a-1. 需求背景格式

需求背景必须说清楚业务背景和业务目标，建议用 2-4 段或短列表，避免空泛。

```markdown
## 2. 需求背景

### 2.1 业务背景
{当前业务场景}：{谁}在做{什么操作}时，{遇到了什么问题}。
{不解决的影响}：导致{什么后果}（效率低/出错多/客户投诉/...）。

### 2.2 业务目标
1. {目标 1}：{做成后用户/业务获得什么结果}
2. {目标 2}：{可感知或可验收的业务收益}
```

示例：
```markdown
### 2.1 业务背景
采购单结算环节：财务人员在核对采购明细和结算金额时，需要手动逐条比对采购单和结算单，
平均每个采购单耗时 15 分钟。当前没有系统化的结算状态追踪，导致结算进度不透明，
经常出现"采购已完成但未结算"的遗漏。

### 2.2 业务目标
1. 结算状态可视化：采购列表和详情页直接展示结算进度，财务无需手动核对
2. 结算操作线上化：发起结算、确认结算在系统内完成，取代线下 Excel 流转
3. 结算遗漏清零：系统自动标记"待结算"的采购单，避免遗漏
```

业务目标编写原则：
- 写**结果**（用户获得什么），不写**产出**（我们做什么功能）
- 正确："财务人员能在 1 分钟内看到所有待结算的采购单"
- 错误："在采购列表增加结算状态筛选功能"（这是功能描述，不是目标）

## 5a-2. 流程图格式

流程图是固定模块，必须包含图片和文字描述。

```markdown
## 3. 流程图

![{流程图名称}](../diagrams/{编号}-flow.png)

### 3.1 流程说明
a. {流程起点、触发条件}
b. {关键判断分支和用户/系统动作}
c. {流程终点、成功/失败结果}
```

简单字段类需求如确实无复杂流程变化，也要生成简化流程图图片，不省略图片：

```markdown
## 3. 流程图

![字段调整流程图](../diagrams/{编号}-flow.png)

### 3.1 流程说明
a. 用户仍按原路径进入页面。
b. 页面在原筛选区/列表区展示本次新增或调整的字段。
c. 查询、导出、详情查看等操作流程保持不变。
```

Mermaid 渲染命令：

```bash
python3 scripts/render_mermaid.py --input diagrams/{编号}-flow.mmd --output diagrams/{编号}-flow.png --type flowchart
```

## 5a-3. 前置非目标与开放问题处理

非目标、开放问题、待确认事项仍然可以在大纲和方案阶段使用，帮助控制范围；但不得作为研发版 PRD 一级章节。

处理规则：
- 非目标：写入 `.tmp/solution_brief.md`，评审沟通时说明；如必须告知研发，放入"需求背景"的一句话中，不单独成章
- 阻塞性开放问题：进入 Step 6 前必须确认，不带入 PRD
- 非阻塞问题：写入 `.tmp/prd_traceability.md`，或在对应详细需求说明中标注"待 XX 确认"

## 5a-4. 功能清单格式规范

功能清单是详细需求的"目录"——列出本次需求涉及的所有改动点，一行一个，不展开细节。

```markdown
## 4. 功能清单

| 模块 | 功能 | 功能点 | 变更类型 |
|------|------|--------|---------|
| 退款 | 退款发起 | 凭证照片上传上限调整 | 修改 |
| 退款 | 退款通知 | 退款发起/成功/失败通知订单手机号 | 新增 |
| 退款 | 退款审批 | 审批表单增加还车时间、退款时间等展示字段 | 新增 |
```

**清单编写原则**：
- **一行一个改动点**：不合并、不展开。"退款凭证照片上限调整"和"退款通知"是两个改动点，必须分两行
- **模块列**：标识业务归属（如"退款""采购""应急收车"），方便按模块筛选和分工
- **功能列**：标识受影响的页面或功能区域（如"退款发起""退款审批""车型配置"）
- **功能点列**：用一句话描述改什么，不写怎么改（怎么改在详细需求里展开）
- **变更类型**：新增 / 修改 / 删除，标识改动性质
- **清单与详细需求的对应关系**：清单每一行，在详细需求中都有对应的表格行展开描述。清单是索引，详细需求是正文

## 5b. 行为层输出规范

**以页面为单位组织输出，使用三列表格**：

详细需求章节按受影响的页面或功能模块组织，每个页面/模块独立成节，使用 `功能点 | 原型 | 说明` 三列 markdown 表格。PRD 初稿阶段原型列可先标注 `原型待补充` 或 `无 UI 变更`；PRD 确认后，在 Step 8 产品原型设计流程中回填原型链接。Step 8 默认通过 Hub 获取 Vue 源码并放入 ProtoHub-mada `preview/` 修改；Hub 不可用时才生成 HTML 原型兜底。

```markdown
## 5. 详细需求

### 5.1 采购单列表页

| 功能点 | 原型 | 说明 |
|--------|------|------|
| 筛选-新增 | | a. 结算状态：多选，枚举：待结算、结算中、已完成，默认不选择 |
| 列表-新增列 | | a. 结算状态：展示当前结算状态 |
| 操作-变更 | | a. 发起结算：对应【结算】权限，结算状态=待结算时展示，点击，打开结算弹框 |

### 5.2 采购单详情页

| 功能点 | 原型 | 说明 |
|--------|------|------|
| 基本信息-新增 | | a. 结算状态：展示当前结算状态 |
| 操作-新增 | | a. 发起结算：对应【结算】权限，结算状态=待结算时展示，点击，打开结算弹框 |
```

**为什么用三列表格**：
- PM 可以直接复制表格到飞书/Confluence，在原型列贴图、在说明列微调描述，**无需二次加工格式**
- 前端开发按功能点行逐行开发，每行对应一个改动
- 产品验收时逐行检查

**说明列的 markdown 格式规范**：

说明列使用结构化 markdown 列表，用 a/b/c → I/II → 1/2/3 嵌套层级表达复杂逻辑。以下按场景分列标准写法：

**筛选字段**：
```markdown
| 功能点 | 原型 | 说明 |
|--------|------|------|
| 筛选-新增 | | a. 物资类型：多选，枚举：固定资产、门店耗材、车辆备品，默认不选择<br>b. 物资简称：模糊查询，汉字、英文、数字、符号，长度<=50 |
```

**列表字段**：
```markdown
| 功能点 | 原型 | 说明 |
|--------|------|------|
| 列表 | | a. 物资编码<br>b. 物资名称<br>c. 操作<br>&emsp;&emsp;I. 编辑：对应【修改】权限，点击，打开编辑弹框 |
```

**新建/编辑表单**：
```markdown
| 功能点 | 原型 | 说明 |
|--------|------|------|
| 新建弹框 | | a. 物资类型：必填，单选，枚举：固定资产、门店耗材、车辆备品，默认不选择<br>b. 物资简称：必填，输入框，汉字、英文、数字、符号，长度<=20<br>c. 操作<br>&emsp;&emsp;I. 取消：关闭弹框<br>&emsp;&emsp;II. 确定：根据物资简称校验唯一性，如果已存在，则toast提示：物资简称重复；如果不存在，则新增物资 |
```

```markdown
| 功能点 | 原型 | 说明 |
|--------|------|------|
| 操作 | | a. 启用：对应【修改】权限，状态为禁用时展示，点击，操作成功，状态切换为启用<br>b. 禁用：对应【修改】权限，状态为启用时展示，点击，操作成功，状态切换为禁用 |
```

**业务流转/条件触发**：
```markdown

|--------|------|------|
| 流程 | | a. 应急收车单状态变更为已到店时，进行消息通知，采购单明细状态变更为已交付，需求单变更为已交付<br>&emsp;&emsp;I. 采购单明细除了已关闭状态外其他全部已交付时，采购单更新成已交付 |
```

**数据带入**：

```markdown
|--------|------|------|
```

| 下拉单选 | 下拉单选，枚举：[选项]/支持搜索 | 支持搜索功能的下拉单选，枚举值：门店基础数据 |
| 日期选择 | 日期选择组件，[查询方式] | 日期选择组件，精准查询 |
| 日期范围 | 日期范围筛选组件，[配置方式] | 日期范围筛选组件，开始日期+结束日期 |

## 5d. 禁止清单

**以下规则适用于所有 PRD 类型。外部对接类 PRD 的例外见下方 5d-1。**

| 禁止 | 原因 | 替代 |
|------|------|------|
| 代码片段 | 需求文档不是技术文档 | 用行为描述 |
| API 接口定义 | 接口设计是开发的决策 | 写"调用XX接口"即可 |
| 数据库表结构 | 限制了工程师的设计空间 | 写字段名和控件类型 |
| 技术选型建议 | PM 没有技术决策的授权 | 不提及技术栈 |
| 代码级拆分 | 代码结构是开发的事 | 写页面和功能点 |
| 算法实现细节 | 工程师的专业领域 | 写业务规则和条件逻辑 |
| CSS/样式代码 | 设计师和前端的事 | 写"展示""隐藏""弹框""toast" |
| "函数""方法""类""接口" | 非业务语言 | 用"操作""按钮""页面""字段" |

### 5d-1. 外部对接类 PRD 的例外

渠道对接、第三方平台接入等外部对接类 PRD，接口字段定义是**双方约定的业务契约**——删掉参数名、枚举值、校验规则，文档就失去对接价值。

| 允许（外部对接类） | 说明 |
|-----------------|------|
| 接口字段表（参数名 + 说明 + 枚举值） | 是业务规则的表达，不是技术实现 |
| 参数校验规则（必填条件、取值范围、格式） | 是产品定义的业务约束 |
| 枚举值完整列表（含中英文映射） | 是双方约定的数据格式 |
| 错误消息定义（msg 内容） | 是用户体验层面的约定 |
| 字段映射表（我方字段 ↔ 对方字段） | 是数据对应关系 |
| 签名/认证规则（业务层面） | 是安全策略的产品约定 |
| 接口名称（如 `qinglu.open.store.base.sync`） | 是双方约定的通信契约标识 |

| 仍然禁止 | 原因 |
|---------|------|
| 代码片段（if/else、循环） | 校验规则用中文描述，不写代码 |
| 内部 API 设计（我方系统内部接口） | 内部接口设计是工程师的决策 |
| HTTP 方法、URL 路径、请求头 | 是技术实现细节 |
| 数据库 DDL、索引设计 | 是技术实现细节 |

### 5d-2. 外部对接类格式示例

**功能模块写法**（按模块组织，每个模块含功能描述 + 操作流程 + 接口契约）：

```markdown
### 5.1 门店信息同步

### 功能描述
门店新增/编辑审核后，将门店信息同步至携程平台。

### 操作流程
a. 门店新增时，首次上线调用【网点新增】接口
b. 门店信息变更审核通过后，自动调用【网点更新】接口
c. 已上线门店支持手动上线/下线操作
d. 未上线门店不展示在平台可租车型列表中

### 接口契约
**【网点新增】API**

| 字段 | 说明 | 枚举/规则 |
|------|------|-----------|
| vender_code | 供应商编码 | 必填，固定值 |
| city_code | 城市编码 | 必填，悟空城市表 |
| name | 门店名称 | 必填 |
| type | 位置类型 | 枚举：普通网点、机场 |
| site_id | 网点ID | 必填，马达门店ID |
| from_time | 营业开始时间 | HH:mm 格式 |
| to_time | 营业结束时间 | HH:mm 格式 |
| minimum | 最小提前预定时间 | 纯数字，单位：小时 |
| night_service_amount | 夜间服务费 | 纯数字，单位：元 |

**校验规则**：
- 门店名称不超过 30 个字符
- 营业时间结束必须大于开始
- 最小提前预定时间范围 0~90 天

**错误消息**：
- 门店已存在：msg "该网点已存在，请勿重复新增"
- 城市编码无效：msg "城市编码不在平台城市表中"
```

**字段映射表写法**：

```markdown
| 我方字段 | 对方字段 | 说明 |
|---------|---------|------|
| 门店ID | site_id | 马达内部 ID 直接映射 |
| 车型ID | car_type_id | 需通过车型映射表转换 |
| 订单状态 | status | 枚举映射：1=已接单，2=已取车，3=已还车 |
```

**错误处理写法**：

```markdown
| 错误场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 接口超时（>5s） | 重试 1 次，仍失败则记录日志 | "操作失败，请稍后重试" |
| 数据不一致 | 以平台数据为准，触发数据补偿任务 | 无用户提示，后台自动处理 |
| 必填字段缺失 | 拦截操作，不允许提交 | "XX字段不能为空" |
```

## Step 8. 产品原型设计（PRD 确认后执行）

PRD 完成并经用户确认后，为有 UI 变更的页面生成或修改原型。默认路径为 Hub 获取 Vue 源码 → 写入 ProtoHub-mada `preview/` → `npm run dev` 预览 → 回填 PRD 原型列；Hub 不可用时，生成高保真 HTML 原型写入 `prototypes/` 目录作为兜底。

### 8-1. 确定需要原型的页面

扫描详细需求的所有 5.X 章节，判断每个页面是否需要原型：

| 条件 | 动作 |
|------|------|
| 有 UI 变更（新增/修改字段、按钮、筛选项） + 映射表有前端仓库和页面路径 | 生成原型 |
| 有 UI 变更 + 映射表前端仓库标注"待确认" | 用脚本 `hub_client.py list-repos --search {关键词}` 搜索定位，成功则生成，失败则标注 `原型待补充（前端仓库未定位）` |
| 无 UI 变更（纯后端逻辑：通知、计算规则、数据对接） | 原型列写 `无 UI 变更` |
| 全新模块（无现有代码） | 基于 PRD 描述和同端组件库从头生成；Web 使用 Element UI， 小程序使用 Vant，标注 `[从头生成]` |

> 原型生成的详细规则见 `references/prototype-rules.md`；ProtoHub 推送、关联检查和自审以远程 `docs/team-guide.md` 为主。

## 需求版本管理格式

每个 PRD 开头必须包含版本管理表格。评审时版本为 `V1.0`；评审通过后根据评审意见修改为 `V1.1`；后续开发、测试提 bug 单导致需求调整时继续按 `V1.2`、`V1.3` 递增。

**修改人来源**：
- 如果来自瓦力需求 → 取 `主产品经理` 字段的姓名部分（去掉后面的数字编号），如 `"郝宇1225623"` → `"郝宇"`
- 如果不是瓦力需求 → 标注 `[待填写]`

```markdown
| 版本 | 日期 | 修改人 | 修改内容 |
|------|------|--------|---------|
| V1.0 | YYYY-MM-DD | 郝宇 | 评审版本 |
```

版本变更规则：
- `V1.0`：评审时使用
- `V1.1`：评审通过后，根据评审意见修改后使用
- `V1.2+`：开发、测试提 bug 单引发需求调整时递增，修改内容写明 bug 单或变更点
- 如果是对已有 PRD 的后续迭代，沿用原 PRD 版本号继续递增，但仍在修改内容中写清评审/bug 单来源

## 验收用例格式

每个 PRD 末尾必须包含验收用例，按页面维度组织。

```markdown
## 6. 验收用例

### 6.1 XX页面验收

| 序号 | 验收项 | 操作步骤 | 预期结果 |
|------|--------|---------|---------|
| 1 | 筛选项展示 | 打开XX列表页，查看筛选区域 | 新增的XX筛选项正确展示，默认不选择 |
| 2 | 筛选功能 | 选择XX筛选项，点击查询 | 列表正确过滤，只展示符合条件的数据 |
| 3 | 列表展示 | 查看列表各列 | 新增列正确展示数据，无数据时显示 `-` |
| 4 | 操作按钮-正常 | 点击XX按钮 | 弹框/页面正确打开，操作成功后toast提示 |
| 5 | 操作按钮-异常 | 不填必填项，点击确定 | toast提示缺少必填项 |
```

**验收用例编写原则**：
- 每个页面的每个功能点至少 1 条验收用例
- 操作步骤写 PM 的实际操作（打开页面、点击按钮、输入数据），不写技术实现
- 预期结果写用户可感知的反馈（展示正确、toast 提示、页面跳转），不写系统内部逻辑
- 覆盖正常路径 + 异常路径（空数据、必填项缺失、边界值）
- 如果涉及状态流转，覆盖每个状态的入口和出口
