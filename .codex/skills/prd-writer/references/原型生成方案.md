# 原型生成能力设计方案

> 历史说明：本文是早期独立 HTML 原型方案的设计记录。当前执行口径以 `SKILL.md` Step 8 和 `references/prototype-rules.md` 为准：需求大纲确认、产品方案/流程图制定、研发版 PRD 确认之后，才进入原型设计；默认通过 Hub 获取 Vue 源码并放入 ProtoHub-mada `preview/` 修改，HTML 仅作为 Hub 不可用时的兜底方案。Web 使用 Element UI，小程序使用 Vant。

---

## 〇、样式对齐原则（最高优先级）

生成原型的样式按照现有前端代码的样式进行生成，前端可复用组件尽量复用。

| 原则 | 说明 |
|------|------|
| **前端代码为准** | 原型样式以实际前端 Vue 源码中的 `<template>` + `<style scoped>` 为唯一标准 |
| **组件复用优先** | Web 优先使用前端代码中已有的 Element UI 组件和共享组件（SearchForm、SearchItem、Table 等）；小程序优先使用 Vant 和项目业务组件 |
| **样式忠实复制** | 从 Hub 获取 Vue 源码后，原样复制组件结构、CSS 类名、scoped styles |
| **无标注装饰** | 原型展示部署后真实状态，不加红色高亮、角标、删除线等标注元素 |

## 一、为什么需要原型生成

PRD 三列表格 `功能点 | 原型 | 说明` 中，原型列当前留空，PM 需要手动截图或画 wireframe 补充。这导致三个问题：

1. **耗时**：每份 PRD 的原型补充需要 30-60 分钟
2. **经常被跳过**：PM 觉得麻烦就不填，开发只能靠想象还原页面
3. **与代码脱节**：PM 画的原型可能和实际页面布局不一致

如果 AI 能从 Hub 前端代码中**还原当前页面**，再**叠加 PRD 变更**，生成可在 ProtoHub-mada `preview/` 中运行的 Vue 原型，PM 和开发就能看到接近真实页面的视觉效果。Hub 不可用时，再生成可直接在浏览器打开的 HTML 原型兜底。

---

## 二、核心链路

```
需求大纲确认 → 产品方案/流程图确认 → 研发版 PRD 确认
  ↓
识别哪些页面受影响 → 映射表定位前端仓库 + 页面路径
  ↓
从 Hub 获取 Vue 组件源码（File:{path} + content:true）
  ↓
AI 解析 Vue 模板结构（搜索表单、表格列、详情字段、弹框）
  ↓
叠加 PRD 变更（新增字段/修改字段/新增按钮）→ 展示部署后的真实状态，不做高亮标注
  ↓
写入 ProtoHub-mada preview Vue 原型；Hub 不可用时生成 HTML 兜底
  ↓
PRD 原型列填写文件链接 + 截图占位
```

## 三、什么时候生成原型

### 生成条件

| 变更类型 | 是否生成 | 原因 |
|---------|---------|------|
| 列表筛选项变更 | ✅ | 有可见的 UI 变化 |
| 列表增加展示字段 | ✅ | 有可见的 UI 变化 |
| 详情页增加信息 | ✅ | 有可见的 UI 变化 |
| 表单增加字段 | ✅ | 有可见的 UI 变化 |
| 操作按钮新增/变更 | ✅ | 有可见的 UI 变化 |
| 附件/上传变更 | ✅ | 有可见的 UI 变化 |
| 审批表单字段变更 | ✅ | 有可见的 UI 变化 |
| 通知/消息触发 | ❌ | 无 UI 变化，后端逻辑 |
| 计算规则（纯后端） | ❌ | 无可视化内容 |
| 外部系统对接 | ❌ | 无 UI 变化 |
| 状态流转（纯后端） | ❌ | 无可视化内容 |

### 判断逻辑

```
IF 功能清单中有"新增""修改"类型的功能点
   AND 该功能点对应的页面在映射表中能找到前端仓库 + 页面路径
THEN 为该页面生成原型

IF 映射表前端仓库标注为"待确认"
THEN 尝试通过 Hub list_repos 定位，定位成功则生成，失败则跳过并标注
```

## 四、五步管线详细设计

### 第一步：确定需要原型的页面

从 PRD 的详细需求章节结构（5.1 XX页面、5.2 XX页面…）提取所有受影响的页面。对每个页面：

1. 在映射表中查找对应的前端仓库和页面路径
2. 如果找到 → 标记为"需要原型"
3. 如果未找到 → 标记为"跳过"（无 UI 变更或无法定位前端代码）

**示例**（PRD #639911）：

```
页面 1：退款发起页 → 映射表命中 views/applyList/components/OrderRefund → 生成原型
页面 2：退款通知 → 无 UI 变更（后端短信） → 跳过
页面 3：退款审批详情 → 映射表命中 views/auditList/components/OrderRefund → 生成原型
```

### 第二步：从 Hub 获取 Vue 源码

对每个需要原型的页面：

**2a. 发现组件文件**

```json
{
  "name": "cypher",
  "arguments": {
    "repo": "LY-MDSCR/mdscr-fe-admin",
    "query": "MATCH (f:File) WHERE f.filePath CONTAINS \"auditList/components/OrderRefund\" RETURN f.filePath LIMIT 20"
  }
}
```

返回示例：
```
src/views/auditList/components/OrderRefund/index.vue
src/views/auditList/components/OrderRefund/components/Pass/index.vue
src/views/auditList/components/OrderRefund/components/Reject/index.vue
src/views/auditList/components/OrderRefund/components/DetailInfo/index.vue
src/views/auditList/components/OrderRefund/components/Detail/index.vue
```

**2b. 获取组件源码**

```json
{
  "name": "context",
  "arguments": {
    "repo": "LY-MDSCR/mdscr-fe-admin",
    "uid": "File:src/views/auditList/components/OrderRefund/components/DetailInfo/index.vue",
    "content": true
  }
}
```

返回 Vue SFC 源码，包含 `<template>` `<script>` `<style>` 三部分。

**2c. 选择关键组件**

一个页面可能有多个子组件。根据 PRD 变更内容选择最相关的：

| PRD 变更类型 | 优先获取的组件 |
|-------------|-------------|
| 列表筛选/列表字段 | `index.vue`（主页面，包含 SearchForm + Table） |
| 详情页字段 | `DetailInfo/index.vue` 或 `Detail/index.vue` |
| 审批表单 | `Pass/index.vue`（通过弹框）、`Reject/index.vue`（驳回弹框） |
| 新建/编辑弹框 | `components/Create/index.vue` 或 `components/Edit/index.vue` |

### 第三步：解析页面结构

**不做形式化的 AST 解析**——利用 AI 的代码理解能力，从 Vue 模板中提取页面骨架。

**提取目标**：

| Vue 组件 | 提取信息 | 示例 |
|---------|---------|------|
| `SearchForm` + `SearchItem` | 筛选项：标签、控件类型 | 订单号（输入框）、退款原因（级联选择）、申请时间（日期范围） |
| `Table` + `el-table-column` | 表格列：标签、宽度 | 订单号、退款原因、原因类型、退款方式 |
| `el-descriptions` + `el-descriptions-item` | 详情字段：标签、列宽 | 订单号(25%)、退款原因(25%)、退款金额(25%) |
| `el-form` + `el-form-item` | 表单字段：标签、必填、校验 | 备注（必填）、执行结果凭证（条件展示） |
| `el-dialog` / `Modal` | 弹框：标题、内容、按钮 | 确认弹框（取消/确定） |
| `el-button` | 按钮：文本、类型、条件 | 通过、驳回 |

**提取示例**（DetailInfo/index.vue）：

```
当前页面骨架：
  el-descriptions（2列布局）
  ├── 订单号 (25%) → 文本展示
  ├── 退款原因 (25%) → 文本展示
  ├── 原因类型 (25%) → 条件文本（订单类/油电费退款）
  ├── 退款方式 (25%) → 文本展示
  ├── 退款金额（元） (25%) → 数字展示
  ├── 退款费用项 (25%) → 标签组展示
  ├── 退款申请凭证 (25%) → 图片组展示（可预览）
  ├── 退款状态 (25%) → 条件展示（审批通过后显示）
  └── 累计人工退款金额 (25%) → 数字展示
```

### 第四步：叠加 PRD 变更

**原型范围原则（重要）**：原型必须展示**完整页面**（让人一眼看出是哪个页面或弹框），但只**修改** PRD 涉及的组件。

| 原则 | 含义 |
|------|------|
| 展示完整页面 | 原型包含页面的主要区域（搜索区、表格区、详情区），让人能辨认出是哪个页面 |
| 只改该改的组件 | PRD 改了哪个组件就修改哪个。其他组件忠实复制 Vue 源码，不做任何调整 |
| 源码忠实度优先 | 未变更字段必须原样复制 Vue 模板结构（标签、顺序、属性），不做任何调整 |
| 禁止设计发挥 | 不添加源码中不存在的组件、布局、装饰元素 |

**页面各区域的处理方式**：

| 区域 | 处理 |
|------|------|
| PRD 变更的组件 | 忠实复制 + 叠加变更，展示部署后真实状态（无标注装饰） |
| 同页面的其他组件 | 忠实复制 Vue 源码，填入模拟数据，不做修改 |
| PRD 未提及的弹框 | 保留触发按钮/入口，不展开弹框内容 |

将 PRD 三列表格中的变更叠加到组件模板上。原型展示的是**部署后的真实页面状态**，不加任何标注装饰（无红色高亮、无角标、无删除线）。变更说明放在底部文档区。

变更类型和视觉处理：

| PRD 变更 | 原型处理 | 视觉标注 |
|---------|---------|---------|
| 新增字段 | 在合理位置插入新字段，与同级元素使用一致的组件和样式 | 无标注（展示部署后状态） |
| 修改字段 | 展示修改后的最终状态 | 无标注 |
| 删除字段 | 从页面上完全移除 | 无标注（不保留灰化版本） |
| 新增按钮 | 在操作区插入按钮，使用一致的 el-button 样式 | 无标注 |
| 新增筛选项 | 在搜索表单中插入，使用 SearchForm 结构 | 无标注 |
| 新增列 | 在表格中插入列，使用一致的 el-table-column | 无标注 |

**位置推断规则**：

- 新增详情字段：放在相关字段附近（如"还车时间"放在"退款时间"旁边）
- 新增筛选项：放在搜索表单末尾
- 新增表格列：放在相关列之后
- 新增按钮：放在操作列末尾

**模拟数据填充**：

- 从 Hub 枚举值中取真实标签（如退款状态用"审核中"而非"状态1"）
- 订单号格式：`ZC` + 年月日 + 7位随机数
- 金额：合理的业务数字（如退款金额 ¥1,280.00）
- 人名：常见中文名（张三、李四）
- 日期：近期日期（2026-05-28）

### 第五步：写入 preview 原型，必要时生成 HTML 兜底

**主路径**：
- 通过 Hub 获取目标 Vue SFC（template + script + style）
- 写入 ProtoHub-mada `preview/src/views/{页面路径}/index.vue`
- Web 使用 Element UI 组件形态，小程序使用 Vant 组件形态
- 用 `cd preview && npm run dev` 构建预览

**兜底路径**：
- Hub 不可用、目标源码无法获取或全新模块缺少参考页面时，生成自包含 HTML 原型
- Web HTML 兜底可使用 Element UI 组件形态；小程序 HTML 兜底用 Vant 组件形态
- 兜底文件放入 `prototypes/`

**HTML 兜底资源示例**：
```html
<link rel="stylesheet" href="https://unpkg.com/element-plus/dist/index.css">
<script src="https://unpkg.com/vue@3/dist/vue.global.prod.js"></script>
<script src="https://unpkg.com/element-plus"></script>
```

**文件命名**：主路径为 `preview/src/views/{页面路径}/index.vue`；HTML 兜底为 `{需求编号}-{页面名称}.html`，如 `639911-退款审批详情.html`

**文件结构**：
```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <title>原型 - {页面名称}（PRD #{需求编号} {需求标题}）</title>
  <!-- CDN 引用，仅用于 HTML 兜底 -->
  <!-- 内联样式：页面布局，禁止变更高亮和标注装饰 -->
</head>
<body>
  <div id="app">
    <!-- 现有页面结构（从 Vue 源码还原） -->
    <!-- 变更后的真实页面状态 -->
    <!-- 变更说明表格 -->
  </div>
  <script>Vue.createApp({}).use(ElementPlus).mount('#app')</script>
</body>
</html>
```

## 五、PRD 中的原型列呈现

### 方案：链接 + 截图占位

```markdown
| 功能点 | 原型 | 说明 |
|--------|------|------|
| 审批详情-新增展示 | [📐 查看原型](prototypes/639911-退款审批详情.html)<br>截图区域：PM 打开链接截图后替换此文本 | a. 还车时间：只读展示...<br>b. 退款时间：只读展示... |
```

**PM 使用流程**：
1. 点击链接 → 打开 preview 页面或 HTML 兜底原型
2. 截图（Mac: Cmd+Shift+4）
3. 将截图贴入 PRD 原型列，替换占位文本

**自动截图（HTML 兜底或 file/localhost 预览时可用）**：
- 已实现 `scripts/screenshot.py`（Playwright Chromium headless 浏览器）
- 策略：等待目标组件渲染完成（`wait_for_selector`）→ 额外 500ms 动画缓冲 → 全页面截图
- 生成 PNG 与 HTML 同名同目录（如 `639911-退款审批详情.html` → `639911-退款审批详情.png`）
- 依赖：`pip3 install playwright && python3 -m playwright install chromium`

### 一个页面有多处变更时

如果同一页面有多个功能点变更，**共用一个原型文件**，在原型中用不同颜色标注不同变更。PRD 中多个功能点行链接到同一个文件。

```markdown
| 功能点 | 原型 | 说明 |
|--------|------|------|
| 筛选-新增 | [📐 查看原型](prototypes/638101-车龄策略配置.html) | a. 车型组：下拉单选... |
| 策略计算-兜底 | [📐 查看原型](prototypes/638101-车龄策略配置.html) | a. 策略计算后车龄<1年... |
```

## 六、降级策略

| 场景 | 降级方案 | 原型列写法 |
|------|---------|-----------|
| Hub 不可用 | 跳过原型生成 | `原型待补充` |
| 前端仓库标注"待确认" | 用 Hub list_repos 搜索，找不到则跳过 | `原型待补充（前端仓库未定位）` |
| Vue 源码过于复杂 | 只提取主组件骨架，子组件用文字说明 | 生成简化版原型 |
| 页面是全新模块（无现有代码） | 基于 PRD 描述和同端组件库从头生成，Web 使用 Element UI，小程序使用 Vant | 标注 `[从头生成]` |
| 非 UI 变更 | 不生成原型 | `无 UI 变更` |

## 八、质量验证

### 原型质量标准

| 维度 | 标准 |
|------|------|
| 布局还原度 | 搜索表单、表格、详情字段的位置和顺序与实际代码一致 |
| 数据真实性 | 使用真实枚举值、合理的业务数据，不用"测试1""数据2" |
| 样式一致性 | 新增内容与原有内容使用同端一致的组件和 props；Web 使用 Element UI，小程序使用 Vant |
| 无标注装饰 | 页面上无红色高亮、角标、删除线等标注元素 |
| 可预览 | 主路径可在 preview dev server 预览；HTML 兜底文件可在浏览器中独立打开 |
| 变更说明 | 文件底部附变更说明表格（el-descriptions），开发可以快速了解改了什么 |

### 验证方法

每次生成原型后，AI 自检：
1. preview 是否可以正常运行；HTML 兜底文件是否可以在浏览器中打开？
2. 所有 PRD 变更是否都在原型中有对应展示？（逐项核对）
3. 现有字段是否从 Vue 源码中正确还原？（与 Hub 返回的源码对比）
4. 模拟数据是否使用了真实枚举值？（与 Hub 枚举对比）
5. 新增内容是否与原有内容使用完全一致的组件和 props？
6. 页面上是否有任何标注装饰？（应为无）
